# ZRule — Architecture

## Tech Stack

| Layer | Technology |
|-------|-----------|
| Runtime | [Bun](https://bun.sh) |
| Backend | [Hono](https://hono.dev) (port 3001) |

| Database | PostgreSQL 17 (via Podman) |

| Frontend | React 19 + Vite 8.1.5 + [TanStack Router](https://tanstack.com/router) + [TanStack Query](https://tanstack.com/query) |
| Rule Editor | [@gorules/jdm-editor](https://github.com/gorules/jdm-editor) |
| Styling | Tailwind CSS v4 + shadcn/ui (base-ui variants) + ReUI Pro blocks |

## Data Model

### Decision Models

| Column | Type | Notes |
|--------|------|-------|
| id | UUID/PK | |
| name | text | |
| slug | text | unique |
| org_id | FK | links to organization (CASCADE delete) |
| graph | JSONB | JDM graph JSON (`{ nodes: [], edges: [] }`) |
| description | text | nullable |
| version | integer | incremented on save |
| created_by | FK | links to user |
| timestamps | | created_at, updated_at, deleted_at (soft delete) |

### Auth Tables (auto-generated by Better Auth)

- **user** — id, name, email, emailVerified, image, createdAt, updatedAt
- **session** — id, userId, token, expiresAt, ipAddress, userAgent, activeOrganizationId
- **account** — id, userId, accountId, providerId, password
- **verification** — id, identifier, token, expires
- **organization** — id, name, slug, logo, metadata, createdAt
- **member** — id, userId, organizationId, role (owner/admin/member)
- **invitation** — id, organizationId, email, role, status, expiresAt

## Key Implementation Patterns

### Dual Runtime (Bun + Node.js)

The server detects the runtime and loads accordingly:

```ts
// apps/server/src/index.ts
const isBun = typeof globalThis.Bun !== "undefined";
if (isBun) {
  Bun.serve({ fetch: app.fetch, port, hostname });
} else {
  const { serve } = await import("@hono/node-server");
  serve({ fetch: app.fetch, port, hostname });
}
```

### Auth Flow

1. Client calls `signIn.email()` → Better Auth handler at `/api/auth/sign-in/email`
2. Server creates session, sets `better-auth.session_token` cookie
3. Subsequent requests carry cookie → `sessionMiddleware` extracts user/session/activeOrganizationId
4. Protected routes use `requireAuth` middleware to enforce authentication + active org

### Organization Scoping

All decision operations are scoped to the user's active organization:
- `sessionMiddleware` reads `activeOrganizationId` from the session
- `requireAuth` rejects requests without an active org
- Decision CRUD filters by `organizationId`

### Vite Proxy

The frontend dev server proxies API requests to avoid CORS issues during development:

```
Client (localhost:5173) → /api/* → Server (localhost:3001)
```

### shadcn/ui (base-ui variants)

Components use `@base-ui/react` — the `render` prop is used instead of `asChild` for composition:

```tsx
<DialogTrigger render={<Button />}>Click me</DialogTrigger>
```

## Frontend Route Structure

| Path | Page | Auth Required |
|------|------|:---:|
| `/` | Redirects to `/dashboard` or `/login` | No |
| `/login` | LoginPage (auth-1 style) | No |
| `/register` | RegisterPage (auth-1 style) | No |
| `/dashboard` | Dashboard (dashboard-1 style) | Yes |
| `/organizations` | Organizations list (settings-1 style) | Yes |
| `/organizations/:id` | Org detail with member management | Yes |
